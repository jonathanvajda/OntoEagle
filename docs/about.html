Edge-first principles for PWA tools (vision + why it matters)

We treat offline as a real operating mode

We design for partial information, deferred resolution, graceful degradation, and explicit uncertainty. Inability to reach external systems isn't a failure state; it's a predictable condition with defined behavior.

What we avoid: tools that collapse the moment connectivity becomes imperfect.

We build edge-capable tools: software that can run, as-is, in a web browser or with a local Node.js runtime (node index.js). That execution model is our “home base.” When we later add cloud hosting, servers, or enterprise deployments, we treat them as helpful accelerators—not prerequisites for correctness.

Why? Because edge-first design gives us something rare: portable power. Teams can evaluate the system without waiting on infrastructure. Users can keep working when networks fail. Architects can reason about behavior without chasing environment-specific side effects. And developers can ship capabilities that scale from a laptop to an organization.

We aim for zero-assumption portability

We write specs so that the core logic stands on its own—without presuming databases, message brokers, service registries, background workers, addressable servers, or a particular topology. That doesn't mean we reject those components. It means we keep them optional and swappable, so different environments can adopt the system without rewriting the "brains."

What we avoid: designs that only work in one deployment shape, or that become expensive to test, demo, or migrate.

We use stochastic tools to make deterministic tools and systems

We design our core capabilities so that the same inputs produce the same outputs. When behavior changes, we want it to be because the inputs changed—not because a hidden service, an ambient cache, or an environment-specific setting quietly shifted.

What we avoid: "it worked yesterday" bugs, production-only failures, and fragile behavior that can't be reproduced locally.

We separate the “brain” from the “body”

We make a clear distinction between:

  • Computation (the reasoning and derivation)
  • State (how results or progress are stored)
  • Orchestration (when and how work is invoked)
  • Integration (how we contact external systems)

In practice, this means the computation stays clean and testable, while storage, scheduling, and integrations plug in as adapters.

What we avoid: tangled systems where changing storage means rewriting logic, or where integrations leak into the core.

We use JSON-LD as the common contract

We treat JSON-LD as our canonical representation for inputs, outputs, configuration, and knowledge structures. Internally, we may optimize—indexes, triples stores, custom graphs—but we keep a firm promise: everything remains losslessly derivable from the JSON-LD form.

What we gain: durable interoperability across tools, teams, and time.

Our "spec test"

We know we're on track when a developer can evaluate, reason about, and run the system using only a browser, local Node.js, and JSON-LD files (at least under the hood).

These constraints aren't about saying "no." They're about making room for clarity: a core that's understandable, testable, and portable—so deployments become choices, not dependencies.
